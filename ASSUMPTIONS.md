## 一、题目留白补全说明

以下九项为题目中未给出精确数值的核心配置项，本系统已做如下假设并实现为可配置项。

### ① 分级审批金额阈值

| 等级 | 金额阈值 | 设定依据 |
|------|----------|----------|
| 一级审批 | < ¥500 | 常规运单单票金额多在 200-800 元区间，500 元为常见的中等金额分界点。低于 500 元的异常通常为小件/低价值货品，一级审批即可覆盖大部分情况 |
| 二级审批 | ≥ ¥500 | 超过 500 元的运单通常为高价值货品或商超大单，需要双重把关降低操作风险 |
| 严重级别 | ≥ ¥1,000 | 单票金额较大，需标记为"serious"级别，在列表中醒目提示 |
| 危险级别 | ≥ ¥5,000 | 大额运单异常需最高警惕，标记为"critical"级别 |

**实现方式**：配置在 `src/lib/config.ts` 的 `APPROVAL_THRESHOLDS` 中，后续可通过系统配置表（system_configs）动态调整，无需修改代码。

### ② 审批超时时长

| 审批阶段 | 超时时长 | 设定依据 |
|----------|----------|----------|
| 一级审批 | 72 小时（3个工作日） | 参考行业标准 SLA：普通工单处理时效为 3 个工作日。72h 给予审批人足够的弹性时间，同时不至于让异常搁置太久 |
| 二级审批 | 48 小时（2个工作日） | 二级审批的工单已经过一级筛选，信息更为清晰，审批决策成本更低，因此时长短于一级 |

**超时处理策略**：一级审批超时 → 自动升级到二级审批；二级审批超时 → 自动驳回（让上报人获得主动权）。

### ③ 重提次数上限

| 配置项 | 数值 | 设定依据 |
|--------|------|----------|
| 最大重提次数 | 3 次 | 遵循"三振出局"原则——3 次驳回说明问题判定分歧较大或上报信息不充分，继续重提只会浪费审批资源 |
| 超上限处理 | 自动升级到二级审批 | 不是直接关闭，而是升级审批层级，由更高级别审批人做最终裁定，避免"一刀切关闭"导致合理诉求被埋没 |

**实现方式**：配置在 `RESUBMIT_CONFIG` 中，可调整。

### ④ 物流异常类型 → 下游动作映射

| 异常类型 | 赔付 | 赔付方向 | 重新发货 | 退货入库 | 回滚库存 | 依据 |
|----------|------|----------|----------|----------|----------|------|
| 丢件 | ✓ | 赔付客户 | ✓ | - | ✓ | 丢件意味着货物灭失，需赔付客户并重发，原运单库存需回滚 |
| 破损 | ✓ | 赔付客户 | ✓ | ✓ | ✓ | 破损货物退回仓库质检，赔付客户并重发 |
| 客户拒收 | - | - | - | ✓ | - | 拒收不产生赔付（客户无损失），货物退回即可 |
| 超时未签收 | ✓ | 赔付客户 | - | - | - | 超时赔付运费/延误费，原货物最终还是到达客户手 |
| 地址错误 | - | - | ✓ | ✓ | - | 纯操作失误，不赔付，退回后按正确地址重发 |

### ⑤ 角色权限划分

| 角色 | 上报工单 | 一级审批 | 二级审批 | 快速放行 | 管理规则 | 管理用户 |
|------|----------|----------|----------|----------|----------|----------|
| reporter（上报人） | ✓ | - | - | - | - | - |
| level1_approver | - | ✓ | - | - | - | - |
| level2_approver | - | - | ✓ | - | - | - |
| qc_supervisor（品控主管） | - | - | - | ✓ | ✓ | - |
| admin（管理员） | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ |

**关键约束**：
- 上报人**不能审批自己提交的工单**（自批自核防护，前后端均校验）
- 一人可兼任多角色（编辑用户时选择对应 role），但系统不推荐同一人既上报又审批
- 快速放行**仅品控主管和管理员**可操作

### ⑥ V2 数据同步频率与一致性策略

| 配置项 | 选择 | 依据 |
|--------|------|------|
| 同步触发方式 | **按需实时同步**（on_demand） | 每次上报异常时实时调用 V2 接口获取最新运单数据，确保异常工单关联的运单信息是最新的 |
| 快照数据过期时间 | 24 小时 | 超过 24 小时的快照数据在详情页标注"数据可能非最新，获取自 XX 时间" |
| 一致性处理策略 | **实时校验 + 快照标注** | 关键操作（上报异常）必须实时校验 V2；展示时使用本地快照 + 明确标注数据来源和时间 |

**V2 服务不可用降级方案**：
1. 接口超时（10s）→ 重试 2 次（间隔 1s、2s）
2. 重试后仍失败 → 基于本地快照数据继续展示，明确标注"V2 服务暂不可用，使用缓存数据（同步于 XX 时间）"
3. V2 恢复后自动切换回实时获取模式

### ⑦ 品控暂扣超时时长

| 配置项 | 数值 | 依据 |
|--------|------|------|
| 品控暂扣超时 | **4 小时** | 品控暂扣意味着货物被物理锁定在仓库中，直接产生仓储成本（压仓费、占用库位、影响周转率）。4 小时是基于以下考虑： |
| | | 1. 一个工作班次内可以完成初步调查和决策 |
| | | 2. 远短于审批超时（72h/48h），体现"货物压仓成本驱动"的设计原则 |
| | | 3. 足够品控主管完成一轮误判复核 |

**为何独立于审批超时**：品控暂扣和审批超时是两个完全不同的业务概念。品控暂扣超时关心的是"货物还要在仓库压多久"，审批超时关心的是"审批人何时处理"。两者的驱动力不同（运营成本 vs. 管理效率），必须独立配置。

**超时后处理**：品控暂扣超时 → 强制升级到二级审批（不等待一级审批完成），确保尽快解除库存锁定。

### ⑧ 品控规则触发阈值

| 规则名称 | 触发字段 | 操作符 | 阈值 | 严重度 | 依据 |
|----------|----------|--------|------|--------|------|
| 数量差异超 10% | quantityDiffPercent | > | 10 | serious | 行业常见容忍上限为 ±5%-10%，超过 10% 需调查 |
| 破损等级 ≥ 2 级 | damageLevel | >= | 2 | serious | 2 级为轻微破损，3 级以上为严重破损 |
| 破损等级 ≥ 3 级 | damageLevel | >= | 3 | critical | 严重破损需强制进入二级审批 |
| 规格偏差超 5% | specDeviation | > | 5 | normal | 按制造业通用公差标准设定 |
| 标签准确度 < 80% | labelAccuracy | < | 80 | normal | 低于 80% 说明标签质量不可接受 |
| 批次不匹配 | batchMatch | = | 0（false） | serious | 混批是严重问题，强制进入二级审批 |

**实现方式**：这些规则存储在 `qc_rules` 表中，通过后台页面（/admin/rules）可随时增删改，**不硬编码**任何触发条件。

### ⑨ 品控主管角色权限边界

| 权限项 | 品控主管 | 一级审批人 | 二级审批人 | 管理员 | 说明 |
|--------|----------|------------|------------|--------|------|
| 操作误判快速放行 | ✓ | - | - | ✓ | 核心权限，仅品控主管可执行 |
| 管理品控规则 | ✓ | - | - | ✓ | 可增删改品控规则配置 |
| 审批一级工单 | - | ✓ | - | ✓ | 审批角色专属 |
| 审批二级工单 | - | - | ✓ | ✓ | 审批角色专属 |
| 上报异常工单 | - | - | - | ✓ | 品控主管不应参与上报 |

**角色不可重叠的设计考量**：
- 品控主管和审批人的权限**不重叠**——品控主管的职责是"品控结果复核"而非"审批业务流程"，两者是决策链的不同环节
- 一个人可以同时被配置为品控主管和审批人（系统技术上不禁止），但业务上不推荐，建议按职责分离原则分配

## 二、向产品经理提出的问题清单

以下是我在理解需求时产生的疑问，体现主动澄清意识：

1. **品控主管误判快速放行后，是否需要通知原始扫描操作人？** 如果操作人不知道自己的判定被推翻，可能会重复扫描同一货物。

2. **多商户/多仓库场景下，审批人是否需要区分仓库？** 即一级审批人是否只能审批自己仓库的工单？跨仓库审批是否允许？

3. **物流异常和品控异常是否可以并存？** 比如一批货在仓库被品控检出异常建工单，后续发货后又发生物流丢件——两个工单是否可以同时存在？我目前的假设是"允许并存"（不同类型不互斥）。

4. **V2 运单在异常处理期间发生金额变更怎么办？** 比如客户申诉后金额被调低，审批中的工单金额是否跟随更新？我目前的设计是"审批时以创建工单时快照的金额为准，详情页标注可能与 V2 不一致"。

5. **超时工单自动升级/驳回后，是否需要通知原审批人？** 当前设计是静默流转，未实现通知机制（可接入钉钉/企业微信）。

6. **赔付记录是否需要对接真实财务系统？** 当前设计只做记录，不做真实支付调用。如果需要对接，需要明确财务系统的接口规范和鉴权方式。

7. **扫描录入是否需要支持批量扫描？** 当前设计为单次录入，如果仓库实际场景是一次扫描多件货物，需要扩展。

8. **审批人离职/禁用后，分配给 TA 的工单如何处理？** 我目前设计了后台任务检测并将工单转交给同角色其他活跃用户，但未实现自动转交的完整逻辑（需要轮询机制）。

## 三、架构假设

1. **单租户假设**：演示环境采用单租户设计，所有用户共享同一仓库。如需支持多租户，需在相关表增加 `warehouseId` 字段并在查询时全局过滤（已在 schema 中预留该字段）。

2. **SQLite 本地开发**：开发阶段使用 SQLite，生产环境切换 Neon/Supabase PostgreSQL。Prisma schema 只需修改 datasource provider。

3. **V2 接口模拟**：目前 V2 接口指向 `https://v2-api.example.com`（占位），实际部署时替换为真实 V2 服务地址。V3 系统在 V2 不可用时基于本地快照降级运行。

4. **定时任务**：超时工单检查通过 `/api/cron` 端点触发，部署到 Vercel 后可配置 Vercel Cron Jobs 定期调用（如每 15 分钟）。本地开发可手动触发或使用 `npm run cron:process`。

5. **AI 集成**：AI 辅助分类和审批建议为可选加分项，当前实现为基于关键词的模拟逻辑。接入真实大模型只需替换 `src/lib/rule-engine.ts` 中的 `aiClassifyException` 和 `aiApprovalSuggestion` 函数。AI 服务不可用时不影响主流程。

## 四、系统间接口契约

### V2 需要提供的接口

| 接口 | 方法 | 路径 | 说明 |
|------|------|------|------|
| 运单校验 + 详情 | GET | `/api/waybills/{waybillNo}/validate` | 校验运单是否存在并返回基本信息 |
| SKU 归属校验 | GET | `/api/waybills/{waybillNo}/sku/{sku}/validate` | 验证 SKU 是否属于该运单 |
| 运单列表同步 | GET | `/api/waybills/sync?page&pageSize&updatedAfter` | 按条件分页查询运单 |
| 异常状态回写 | POST | `/api/waybills/{waybillNo}/exception-notify` | 通知 V2 运单存在未关闭异常（加分项） |
| 健康检查 | GET | `/api/health` | V2 服务可用性探测 |

### 鉴权方式

- V3 → V2：请求头携带 `X-API-Key` 进行服务间鉴权
- V2 → V3（如做了回写）：同理携带 API Key

### 接口响应格式

```json
{
  "success": true,
  "data": {
    "waybillNo": "WB20240101001",
    "senderName": "张三",
    "receiverName": "李四",
    "declaredAmount": 500.00,
    "status": "in_transit",
    "skuInfo": [{ "sku": "SKU-001", "name": "商品A", "qty": 5 }],
    "warehouseId": "WH-001"
  }
}
```
